# 섹션 01. JPA 소개
## 01. SQL 중심적인 개발의 문제점
우리가 개발을 할 때 보통 `객체 지향 언어`로 개발을 하게 됨
- 그리고 `DB`를 다룰 때는 `관계형 DB`를 많이 사용하게 됨  
### 현재는 `객체`를 만들어 해당 오브젝트를 `관계형 DB`에 보관/관리하는 시대
`객체`를 생성해 `DB`에 `저장/조회/수정/삭제(CRUD)` 할 때 수 많은 `SQL`을 사용하게 됨
	- 결국 이러한 부분들이 지속적으로 반복되기에 `SQL 중심적 개발`에 문제점이 발생  
<br/>

### 패러다임의 불일치
하지만 우리는 `관계형 DB`를 사용하고, 또 통신해야 하기 때문에 `SQL 의존적 개발`을 피하기가 어려움
- 즉, 현재 주어진 상황에서 식별한 문제점을 해결하는데 어려움이 있음  
<br/>

### 객체와 RDB 의 차이
1. `상속`: `RDB`에는 기본적으로 `객체`에서 말하는 `상속관계`가 존재하지 않음
2. `연관관계`: `객체의 경우 참조`를 , `RDB 의 경우 FK(외래 키)`를 사용  
<br/>

### 점점 늘어나는 매핑 작업
기존 매핑 작업도 많은데 좀 더 `객체`스럽게 모델링을 하려고 하면 `매핑 작업은 더 늘어`난다.
- 많은 개발자들이 `'DB 에도 자바 컬렉션처럼 객체를 저장 할 수 없을까?'`라 생각 했을 것이다.
- 그래서 등장한 것이 바로 `JPA(Java Persistence API)`!  
<br/><br/>

## 02. JPA 소개
### JPA (Java Persistence API)
자바의 `ORM 기술 표준`  
- `Application`과 `JDBC` 사이에서 동작
- `JPA`는 객체를 저장해 달라는 요청을 받으면 해당 `객체를 분석`, `SQL 작성`해서 `JDBC API`를 호출 해 `DB에 쿼리까지 날려줌`
	- 더해서 `DB`에서 가져온 `Result Set`을 매핑까지 해준다.
	- 자바 컬랙션처럼 `RDB`를 다룰 수 있게 되며 `패러다임 불일치`를 해결하게 되었다!  
<br/>

### ORM (object-relationship mapping)
`객체 관계 매핑`을 말하며, `객체`와 `RDB` 사이에서 매핑을 해준다고 이해하면 되겠다.
- `객체는 객체대로`, `RDB 는 RDB 대로` 설계가 가능하게 한다.
- 대중적인 프로그래밍 언어에는 대부분 `ORM 기술`이 존재한다.
- `ORM`은 `객체`와 `RDB` 두 기둥 위에 있는 기술이라 할 수 있음  
<br/>

### JPA 꼭 사용해야 할까?
이미 자바 진영의 `표준 ORM`인 것이 크겠지만 아래의 장점을 보면 `"사용하겠습니다!"`라는 말이 절로 나올 것이다.
- `JPA 사용 시 장점`
	- `SQL 중심적 개발`에서  `객체 중심 개발`이 가능해짐
	- `생산성`과 `유지보수성`이 훨씬 늘어남
	- `패러다임 불일치`를 해결 할 수 있음
	- `성능` 이점이 있음
